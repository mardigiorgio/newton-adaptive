### Old-style kwargs (NOT present in current constructor -- callers will break)

`scripts/demos/step_quadruped.py:42-48` and `scripts/control/cenic_step_anymal_walk.py:134-145`
and `scripts/control/cenic_double_pendulum_lqr.py:284-291` pass:

```python
solver = newton.solvers.SolverMuJoCoCENIC(
    model,
    tol=1e-3,
    dt_init=...,   # does NOT exist -- should be dt_inner_init
    dt_min=...,    # does NOT exist -- should be dt_inner_min
    dt_max=...,    # does NOT exist -- should be dt_inner_max
)
```

These three callers use the old parameter names from a prior API version.
Because `SolverMuJoCoCENIC.__init__` accepts `**kwargs` and forwards them to
`SolverMuJoCo`, these kwargs are silently swallowed (treated as `SolverMuJoCo`
options), and the CENIC controller uses the default `dt_inner_*` values instead
of the caller's intent.  This is a latent bug in those scripts.

## Risks for the refactor

1. **Old-style constructor kwargs (`dt_init`, `dt_min`, `dt_max`) silently ignored.**
   `scripts/demos/step_quadruped.py`, `scripts/control/cenic_step_anymal_walk.py`,
   and `scripts/control/cenic_double_pendulum_lqr.py` all pass `dt_init=`,
   `dt_min=`, `dt_max=` to `SolverMuJoCoCENIC`. The current constructor does not
   have those params; they fall through `**kwargs` to `SolverMuJoCo` which also
   ignores them silently. The callers therefore run with the default `dt_inner_*`
   values, not their intended values.  The refactor must not make this worse, but
   ideally should add a deprecation warning or raise on unknown kwargs.

2. **`solver.step()` with `contacts=None`** is called in three control scripts.
   `SolverMuJoCoCENIC.step()` ignores its `contacts` argument (generates contacts
   internally), so `None` is fine.  The `step_dt` path is the recommended API
   (per CLAUDE.md), but `step()` is still used in old-style inner loops that
   re-implement the boundary check manually with `sim_time.numpy()`.  Those loops
   violate the CLAUDE.md hot-path rule (device sync inside inner loop) but that
   is a caller issue, not a refactor concern.

3. **`solver.sim_time` passed as raw `wp.array` to a kernel.**
   `scripts/control/cenic_double_pendulum_lqr.py:318,384` passes `solver.sim_time`
   directly as a kernel input (`wp.launch(..., inputs=[..., solver.sim_time, ...])`).
   If the refactor replaces `_sim_time` with a property that returns a differently-
   shaped array, the kernel will silently get wrong data.  The array must remain
   `shape=[world_count], dtype=float32`.

4. **`solver.contacts` must return the live internal `Contacts` buffer.**
   Three demos reuse `solver.contacts` in the render loop (`viewer.log_contacts`).
   If the refactor allocates a new `Contacts` object per step, those references
   will become stale.  The buffer must be allocated once and reused.

5. **`_tol`, `_dt`, `_dt_max` direct access.**  These are used in display strings
   in seven callers.  Adding public `tol`, `dt_max` properties (alongside the
   existing `dt` property) would let callers migrate cleanly.  The refactor should
   keep `_tol` and `_dt_max` accessible (or add the properties) to avoid breaking
   the banners.

6. **`isinstance(solver, SolverMuJoCo)` is not checked anywhere** in the scanned
   files, so making CENIC a non-subclass shim does not break any isinstance guard
   in this repo.  However, external code or future tests could depend on it.
